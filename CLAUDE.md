# 옷장요정(Closet Fairy) 다이어그램 · 문서 레포

## 1. 목적
- 이 레포는 **"옷장요정" 캡스톤 프로젝트에 필요한 모든 다이어그램을 작성하고 문서로 정제**하기 위한 곳이다.
- 기술 스택·실행 방법은 [README](./README.md) 참고 — Astro Starlight + PlantUML.
- **다루는 문서**
  1. Use-Case Diagram — 시스템 유스케이스 정의
  2. SRS (Software Requirements Specification) — 소프트웨어 요구사항 명세

## 2. 주임무
- 사용자가 **문서·서면 자료를 전달하며 다이어그램 생성을 요청**하는 것이 이 레포의 주임무다.

## 3. 원칙

### 3.1 다이어그램 종류 확인
- 🚨 사용자가 **어떤 다이어그램인지 명시하지 않았다면**, 바로 작업에 착수하지 말고 **먼저 확인 질문**을 한다.

### 3.2 협업 태도
- 사용자를 **명령을 따르는 에이전트가 아니라, 동일한 권한을 가진 팀원**으로 대한다.
- 명확한 지시는 따르되, 단순 대화 중이거나 의견을 구하는 상황이라면 **나도 적극적으로 의견을 낸다.**

### 3.3 요구사항 처리
- 부족하거나 미완성인 요구사항이 많이 주어질 수 있다. **빈자리를 임의로 추론해 채우지 않는다.**
- 모호한 부분은 **하나도 빠짐없이 사용자에게 반복 질문**하여 정확한 요구사항을 확보한다.
- 🚨 **잘못된 요구사항이 반영되는 것은 모호한 요구사항이 남는 것보다 훨씬 치명적이다.** 절대 있어서는 안 된다. (확신이 없으면 추론하지 말고 질문한다.)

### 3.4 추후 논의가 필요한 부분
- 사용자가 **"추후 논의가 필요하다"는 뉘앙스**로 이야기하면, 해당 부분은 **다이어그램에 억지로 그리지 않고 비워둔다.**
- 논의사항 자체는 **사용자가 직접 작성한다(3.10).** Claude는 채우지 않으며, 적을 내용이 없으면 그 문단을 두지 않는다. 사용자가 요청할 때만 채운다.

### 3.5 요구사항 구조 · 위계
- 요구사항은 **간트차트(Gantt chart) 형태**의 구조로 주어진다. 사용자는 (a) 간트차트 자료를 **한꺼번에** 전달하거나, (b) `7.1.2.1`처럼 **위계(번호)** 와 함께 특정 요구사항 추가를 요청한다.
- 🚨 **위계가 주어지지 않으면 섣불리 문서를 생성하지 말고, 먼저 위계를 묻는다.** 사용자가 위계 개념을 이해하지 못하면 **"어느 문서의 하위로 생성할지"** 를 더 풀어서 쉽게 묻는다.
- 위계(번호)는 **배치·식별 용도로만 쓰고, 생성되는 문서 안에는 표시하지 않는다.**

### 3.6 기능 단위가 아닌 항목
- 간트차트 항목 중 일부는 **기능의 단위가 아닐 수 있다** (예: `1.1.1 [제안서 작성]` — 위계상 존재하지만 기능 구현과는 무관).
- 이런 항목은 **다이어그램·문서로 생성하지 않는다.**
- 대신 **생성하지 않은 항목들을 표로 정리해 사용자에게 보고**한다. (채팅 보고로 충분하며, 레포 문서에는 남기지 않는다.)
- **유스케이스는 "기능"을 그리는 도구이지 UI를 그리는 도구가 아니다.** UI/화면 항목은 별도 유스케이스로 만들지 않는다(예: UI/UX 시스템 전체 제외). UI를 다뤄야 한다면 그 UI가 **담당하는 기능**으로 표현한다.

### 3.7 문서 작성 순서
- 먼저 **Use-Case Diagram**을 확정하고, 이를 **기반으로 SRS**를 정립한다.
- Use-Case Diagram이 SRS의 근거이므로, **유스케이스가 변경되면 SRS도 함께 점검**한다.
- SRS의 구체 **양식/템플릿은 SRS 단계에서 함께 정한다** (그 단계는 다른 담당자가 진행할 수 있음).

### 3.8 위계 페이지 구조 · 부모 페이지 다이어그램
- **위계 = 페이지, 하위 항목 = 중첩 폴더**(자식 페이지). 사이드바는 수동으로 구성한다(3.13).
- **자식(말단) 페이지**: 해당 기능의 유스케이스를 그대로 상세히 그린다.
- **부모 페이지**: 자식을 전부 가져와 복제하지도(중복·동기화 부담), 빈 직사각형으로 비우지도 않는다. **직속 자식 각각을 요약 유스케이스(오발 1개)로** 그린 **개요(overview) 다이어그램**을 만든다 — 부모는 자식들의 "지도", 자식은 "실제 그림".
  - 부모가 **직접 가진 말단 기능**이 있으면 일반 유스케이스로, 하위 그룹은 요약 유스케이스로 **혼합**한다.
  - **순수 그룹핑 부모**면 자식 요약 오발 + (선택) 각 오발에서 자식 페이지로 하이퍼링크(PlantUML `[[링크]]`).
  - 추상화 레벨을 페이지마다 일관되게: **부모 = 요약 수준, 자식 = 사용자 목표 수준.**
- **부모 개요의 액터**: 기본값으로 **직속 자식들의 액터 합집합(중복 제거)**을 표시한다. 특정 다이어그램이 과밀하면 그때 추린다.
- **위계·페이지 구조는 고정이 아니다.** 굳이 나눌 필요 없는 유스케이스는 합치고, 요구사항이 복잡해지면 그때 나눈다. 구조 유지 자체를 목적으로 삼지 않는다(원칙 4.4·4.5와 함께 판단). 문서 수정 비용은 AI에 위탁할 수 있으니 유연하게 재구성한다.

### 3.9 액터 · 시스템 경계
- **액터 목록은 Claude가 작성(제안)하고, 사용자가 이후 조정**한다. (처음부터 완벽할 필요 없이, 초안 → 조정 방식)
- **시스템 경계(`rectangle`)는 그 페이지가 속한 기능 단위(서브시스템)로 한다.** 전체 서비스 이름("옷장요정")을 모든 페이지의 공통 경계로 쓰지 않는다. 같은 영역(챕터)의 페이지들은 같은 서브시스템 경계를 공유하고, 위계는 "어떤 유스케이스가 어느 페이지에 오느냐"로 표현한다.

### 3.10 페이지 문서 형식 (통일)
모든 문서 페이지는 아래 섹션을 이 순서로 작성한다. 1~3은 항상 두고, 4(논의사항)는 적을 내용이 있을 때만 둔다. (이전의 임시 섹션 구성을 대체. SRS는 일단 제외한다.)
1. **개요** — 이 페이지가 다루는 기능을 간략히 설명한다.
2. **요구사항** — 절차대로 정제하여 **빠짐없이 엄밀하게** 줄글로 서술한다. 복잡하면 길어져도 된다.
3. **유스케이스 다이어그램** — 요구사항에 대응하는 PlantUML 다이어그램을 채운다.
4. **정해야 하는 논의사항** — 모호했던/미정 사항을 적는 영역. **사용자가 직접 작성하는 영역이므로 Claude는 채우지 않는다.** 적을 내용이 있을 때만 이 문단을 두고, **비어 있으면 제목을 포함해 문단째 두지 않는다.** 사용자가 요청하면 채운다.

각 섹션은 본문에서 `##`(H2)로 쓰고, 하위 항목은 `###` 이하 헤딩을 자유롭게 쓴다. (frontmatter `title`이 H1이므로 본문 최상위는 H2.)

### 3.11 단일 출처(SSOT) — 부모/자식 중복 방지
- **글(요구사항·논의)은 그것을 "소유"하는 가장 구체적인 한 페이지에만** 적고, 다른 페이지는 복사하지 말고 **링크로 참조**한다.
  - 특정 자식에만 해당 → 그 **자식 페이지에만**.
  - 여러 자식 공통이거나 상위 차원 → **부모 페이지에만**(자식은 "상위 참고" 링크).
  - 어디에 속할지 애매하면 **더 구체적인(자식) 쪽**에 두고 부모는 링크.
- **유스케이스 다이어그램은 예외(중복 아님)** — 부모=요약 / 자식=상세(3.8)라 추상화 레벨 차이이지 복붙이 아니다.
- 목적: 빈번한 수정에도 **한 곳만 고치면 되고**, 문서 간 모순이 생기지 않게.

### 3.12 문서 문체 · 표현
- 불필요하게 어렵거나 생소한 단어를 쓰지 않는다.
- 특히 사용례가 어색한 외래어나 이른바 "판교체"(과한 IT 외래어 말투)를 지양한다.
- **요구사항**은 꼭 언급해야 하는 개발 용어를 빼면 비개발자도 이해할 수 있게 쓴다. 개발 용어를 쓸 때는 쉬운 말로 풀어 준다.
- **줄표(—)를 쓰지 않는다.** 영어권에서는 흔하지만 한국어에서 어색한 표기나 문장 형식을 피하고, 줄표 없이도 매끄럽게 읽히도록 문장을 다듬는다.
- "**불필요하게**" 어려운 것만 지양한다. 내용상 꼭 필요한 어려움이나 개발 용어는 **억지로 풀어 해치지 않는다.** 강제로 분해하면 오히려 읽기 어려워지고 정확성·전문성을 잃을 수 있다. 판단 기준은 "이 어려움이 내용 때문인가, 표현 선택 때문인가".
- **모델링 방식에 대한 메타 설명을 문서에 쓰지 않는다.** "왜 이렇게 그렸는지", "무엇이 유스케이스가 아닌지", "나중에 분리한다" 같은 설명은 빼고, 사이트에는 결론(요구사항)만 적는다.

### 3.13 사이드바 · 카테고리 표시 (수동 구성)
- 사이드바는 autogenerate가 아니라 **수동 구성**(`astro.config.mjs`의 `sidebar`)으로 관리한다.
- **카테고리(그룹) 라벨은 숫자(2, 5-1 등)가 아니라 기능명**으로 한다. "유스케이스" 같은 래퍼 그룹을 두지 않고, 각 영역을 최상위 그룹으로 올린다.
- 각 폴더의 **상위(개요) 페이지는 사이드바에서 "개요"로** 표시한다(`{ label: '개요', slug: 'use-cases/...' }`). 페이지 제목(H1)은 기능명을 그대로 둔다. 그룹 라벨과 이름이 겹치지 않게 하기 위함이다.
- 내부 링크는 `slug`로 건다. 빌드 시 검증되고 현재 페이지 활성 표시도 맞는다.
- **페이지를 추가하거나 삭제하면 이 `sidebar` 목록도 함께 갱신한다.**

### 3.14 전체 시스템 다이어그램 동기화
- 최상단 **"전체 시스템"** 페이지는 모든 영역의 유스케이스 다이어그램을 하나로 취합한 것이다. 이 페이지는 **개요와 유스케이스 다이어그램 두 문단만** 둔다.
- 이 다이어그램에 한해 **하위 항목을 요약 오발 하나로 뭉뚱그리지 않고, 모든 유스케이스를 생략 없이 싣는다.** (3.8의 "부모는 요약" 규칙의 예외다. 각 유스케이스의 상세 요구사항은 해당 영역 페이지에서 다룬다.)
- 🚨 **다른 페이지의 유스케이스 다이어그램이 바뀌면(유스케이스 추가·삭제·이름 변경, 액터·관계 변경 등) 항상 "전체 시스템" 다이어그램도 함께 갱신한다.**
- 보관용 `(Outdated)` 페이지는 취합 대상이 아니다.

## 4. 유스케이스 다이어그램 작성·검수 원칙
권위 있는 자료(OMG/UML 정의, Cockburn, Fowler, Bittner, Scott Ambler, Visual Paradigm, uml-diagrams.org 등) 조사로 정립했다. 캡스톤은 화려함보다 기초가 우선이므로 이 원칙을 고수한다.

### 4.1 유스케이스의 본질
- 유스케이스 하나는 **액터에게 관찰 가능한 가치를 주는 완결된 목표 단위**다. 끝까지 끝났을 때 액터에게 의미 있는 결과가 남아야 한다. 내부 처리 단계나 부분 동작은 유스케이스가 아니다.
- 판별은 "누가 시작하느냐"가 아니라 **"액터에게 관찰 가능한 완결 가치가 있느냐"**다. (UML 2.5는 액터 개시 요건을 없앴다. 스케줄러가 시작해도 액터에게 가치를 주면 유스케이스가 될 수 있다.)
- 판별 질문(커피 한 잔 테스트): "이 일을 끝낸 뒤 1차 액터가 만족하고 자리를 떠날 수 있는가." 그렇다면 유스케이스, 아니면 단계다. (로그인은 대개 단계, "~검증/~저장"은 단계.)
- 이름은 **동사+목적어의 액터 목표** 형태로 짓는다(예: 옷 등록, 코디 추천 받기). "~자동 처리", "~검증", "~저장" 같은 내부 구현 냄새 이름은 피한다. 도메인 용어를 쓰고 기술 용어는 피한다.

### 4.2 액터와 시스템 경계
- 액터는 항상 **시스템 외부의 역할**이다. 설계 대상의 내부 컴포넌트(서버, DB, 내부 모듈)는 액터가 아니다. 진짜 외부 시스템(제3자 서비스, 외부 결제, 외부 API)일 때만 액터(보통 2차 액터)다. 기준은 "기계냐 사람이냐"가 아니라 **"내부냐 외부냐"**.
- 절차: 시스템 경계(무엇을 우리가 구현·변경하는가)를 먼저 확정하고, 경계 밖에서 능동적으로 행위하는 것만 액터로 둔다. "코드를 변경 못 하고 우리 배포 단위에 없으면 외부".
- 액터는 역할로 명명한다(개인 이름 금지). 액터끼리는 상호작용하지 않는다. 액터는 경계 박스 밖, 유스케이스는 안에 둔다.

### 4.3 관계(include / extend / generalization)
- **include**는 필수 공통 동작의 재사용에만. 베이스는 included 없이 완성되지 않는다. 화살표는 베이스에서 included 방향.
- **extend**는 선택·조건부 부가 동작에만. 베이스는 확장이 없어도 완결된다. 화살표는 확장에서 베이스 방향. 가드 조건은 다이어그램이 아니라 텍스트 명세에 적는다.
- **generalization**은 is-a(특수화)에만. 화살표는 자식에서 부모 방향. 표기는 실선과 속찬 삼각형 화살촉(include/extend의 점선과 열린 화살촉과 다름).
- 관계로 **시간 순서나 단계 흐름을 표현하지 않는다.** include/extend를 "A 다음 B" 순서 그리는 데 쓰면 안 된다.
- 관계 구분에 과하게 매달리지 말 것. 먼저 완결된 목표 유스케이스부터 그리고, 더 명료해질 때만 관계를 추가한다. include는 보통 1단계 깊이로 제한한다.

### 4.4 안티패턴 (피할 것)
- **내부 처리·구현 단계(검증, 저장, 계산, "서버 자동 처리")를 타원으로 올리지 않는다.** 본문 시나리오나 시퀀스/액티비티 다이어그램으로 내린다.
- **기능 분해 금지.** 한 가지 일에는 유스케이스 하나면 충분하다. 잘게 쪼갠 하위 기능은 그 유스케이스의 대안/하위 흐름이다. (한 액터를 작은 타원들이 빙 두르면 분해 징후.)
- **CRUD/엔드포인트를 객체마다 타원과 1:1로 기계적으로 나열하지 않는다.** 같은 액터의 같은 객체 CRUD는 기본적으로 묶는다. 분리는 (a) 액터가 실제 다르고 (b) 독립적으로 시작되며 (c) 사전·사후조건이 다를 때만. 권한 차이만으로는 분리하지 않는다.
- **순서·시간·반복·제어 흐름을 표현하지 않는다**(그건 액티비티/시퀀스/상태 다이어그램의 일).
- **연결선은 실제 상호작용을 반영한다.** 액터가 그 유스케이스를 직접 개시하거나 결과의 이해관계자일 때만 선을 긋는다. 자동 하위 유스케이스에 액터를 직접 잇지 않는다. 연관선에 데이터 흐름 화살촉을 붙이지 않는다.

### 4.5 세분도와 텍스트 명세
- 유스케이스는 **사용자 목표 수준(Cockburn sea level)**으로 잡는다. 그보다 낮은 하위 기능·검증·모듈 수준은 주 타원으로 올리지 않는다.
- 한 장의 타원이 5~9개를 크게 넘으면 과분해 신호다(합격선이 아니라 경고). 위계 분할 레포에서는 부모 개요에 직속 자식만 싣는다.
- **진짜 가치는 텍스트 시나리오에 있다.** 각 유스케이스는 트리거·전제조건·사후조건·기본 흐름·대안 흐름을 텍스트로 명세한다. 다이어그램은 그 목차다.
- 커버리지도 본다. 고립된 액터가 없는가, 주요 사용자 목표가 모두 있는가, 외부 2차 액터가 빠지지 않았는가.

### 4.6 검수 체크리스트
1. 모든 타원이 사용자 목표 레벨인가. "~검증/~저장/서버 자동 처리" 같은 내부 처리가 타원으로 올라오지 않았나(커피 한 잔 테스트).
2. 액터가 모두 시스템 외부의 능동 역할인가. 내부 서버/DB를 액터로 두지 않았나.
3. include/extend로 "로그인 다음 검색 다음 결제" 같은 시간 순서를 그리지 않았나.
4. 같은 객체 CRUD를 타원 여러 개로 나열하지 않고 적절히 묶었나(분리는 다른 액터 + 독립 개시 + 다른 사전·사후조건일 때만, 권한 차이만으로는 안 됨).
5. 액터-유스케이스 선이 실제 개시/이해관계를 반영하나. 자동 하위 유스케이스에 액터를 직접 잇지 않았나.
6. include는 필수 공통, extend는 선택·조건, generalization은 is-a로 맞게 썼나.
7. 표기가 맞나(include/extend는 점선과 열린 화살촉, generalization은 실선과 속찬 삼각형). 방향 반전이나 순환 include 없나.
8. 한 액터를 작은 타원들이 빙 두르는 기능 분해 징후가 없나.
9. 액터는 박스 밖, 유스케이스는 박스 안인가.
10. 고립된 액터가 없나. 주요 사용자 목표와 외부 2차 액터가 빠지지 않았나.
11. 각 유스케이스에 트리거·전제·사후·기본/대안 흐름이 텍스트로 있나(가드는 다이어그램 아닌 명세에).
12. 한 장 타원이 5~9개를 크게 넘지 않나. 부모 개요엔 직속 자식만 실었나.
13. 가독성(크기·간격·교차)이 과하지 않나(감점 아닌 소프트 권고. PlantUML 자동 레이아웃이면 요소를 줄이고 분할).
14. 평가/제출용이면 표준 표기를 정확히 지키고 include/extend는 1단계 깊이로 했나.
