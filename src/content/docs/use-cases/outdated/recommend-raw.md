## 기능 요구사항

### 실시간 날씨 및 TPO 데이터 룰 필터링

| 항목 | 내용 |
| --- | --- |
| 요구사항 유형 | 기능 |
| 요구사항 개요 | 외부 기상 API로부터 수집한 실시간 날씨·기온 데이터를 미리 정의된 룰 기반 필터링 엔진을 통해 현재 환경에 부적합한 의류를 추천 후보군에서 제거한다. |
| 요구사항 내용 | **1.1** 시스템은 추천 요청 시점에 사용자 위치 정보를 기반으로 외부 기상 Open API로부터 현재 온도, 날씨 상태, 시간대별 온도 변화를 실시간 수집한다.<br>**1.2** 수집된 환경 데이터를 기반으로 Rule-Based Filtering 엔진을 가동하여 다음 조건에 따라 옷장 의류를 필터링한다: 기온 28도 이상 시 패딩·두꺼운 아우터 제거 / 체감온도 15도 이하 시 아우터 카테고리 우선 포함 / 강수 감지 시 방수 소재 의류 우선 / 여름철 두꺼운 니트류 제거.<br>**1.3** TPO(데이트, 출근, 결혼식 하객, 운동 등) 정보는 LLM 프롬프트로 컨텍스트로 전달하여 환경 정보(온도, 날씨)와 조합하여 추천 컨텍스트를 구성한다.<br>**1.4** 유저 선호 스타일에 따라 필터링 우선순위를 보조적으로 조정한다. 예: 미니멀 스타일 선호 시 과한 패턴 의류의 추천 우선순위를 하향 조정한다.<br>**1.5** 전처리 완료된 의류 후보 배열과 컨텍스트 JSON은 Dynamic Prompt Injection 단계로 전달된다. |
| 입력 데이터 | 사용자 위치 정보, 외부 기상 API 응답(온도·날씨), 사용자 TPO 텍스트 입력, 옷장 의류 등록, 외출 시간대, 체질, 선호 스타일 |
| 출력 데이터 | 필터링이 완료된 추천 후보 의류 배열 및 전처리된 컨텍스트 JSON |
| 입출력 유형 | 외부 API 수신 및 내부 데이터 연산 |
| 데이터 파일 유형 | 내부 파일(룰 딕셔너리 설정 파일, 필터링 결과 JSON) |
| 중요도 | 상 |
| 위험도 | 중 |
| 품질속성 | 기능 적합성, 신뢰성, 상호운용성 |
| 평가 방법 | 1. 기온 28도 이상 조건의 가상 날씨 데이터를 주입하여 패딩류가 추천 후보에서 정상 제거되는지 필터링 결과 배열을 검증한다.<br>2. 강수 상태 날씨 데이터 주입 시 방수 소재 의류가 우선 포함되는지 출력 배열을 확인한다.<br>3. 외부 기상 API 응답 실패 시 시스템이 에러 없이 기본값(현재 계절 평균 기온·맑음)으로 대체 처리하는지 확인한다. |

### Dynamic Prompt Injection 및 다중 코디 생성

| 항목 | 내용 |
| --- | --- |
| 요구사항 유형 | 기능 |
| 요구사항 개요 | 유저 선호 점수 DB를 조회하여 dynamic_preferences 데이터를 추출하고, 이를 입력 프롬프트와 결합하여 다중 코디 세트(3\~4개)와 자연어 추천 사유를 동시 생성한다. |
| 요구사항 내용 | **1.1** 시스템은 추천 파이프라인 가동 시 해당 유저 ID의 스코어 테이블로부터 선호/기피 스타일 및 색상 인덱스를 정렬 조회한다.<br>**1.2** 양수(+) 점수 상위 항목은 선호(highly_preferred)로, 음수(-) 점수 항목은 기피(disliked)로 분류하여 dynamic_preferences JSON 프롬프트 컨텍스트를 생성 및 삽입한다.<br>**1.3** 메인 코디 생성 LLM은 3\~4개의 서로 독립적인 코디네이션 후보 세트를 동시 리턴해야 한다. 각 세트는 아우터·상의·하의·신발·양말·악세서리 등 착용 가능한 아이템 매핑 목록을 포함해야 한다.<br>**1.4** [LLM 추천 설명 생성] 각 코디 세트에 대해 날씨·체감온도 적합성, TPO 적합성, 사용자 선호 스타일 반영 여부를 근거로 한 구체적인 자연어 추천 사유(reason 필드)를 반드시 동시 생성해야 한다.<br>**1.5** [자연어 수정 재추천] 사용자가 자연어 수정 요청(예: "더 격식있는 스타일")을 입력한 경우, 해당 요청 텍스트를 프롬프트 피드백으로 포함하여 새로운 코디 후보를 재생성한다. 수정 요청은 Attribute-level 피드백으로 처리되어 가중치 시스템에도 연계 반영된다.<br>**1.6** 가중치 Time Decay는 피드백 발생 시점과 분리하여 월 1회 배치 작업으로 전체 유저 점수에 일괄 적용한다.<br>**1.7** [활용/탐색 조건] 다중 코디 세트(3\~4개) 중 2\~3개는 선호 상위 스타일을 활용하고, 1개는 최근 30일 내 미추천 또는 선호도가 낮은 스타일(날씨/TPO를 만족하는 조건 충족)을 탐색하여 구성한다. 단, 음수(-) 점수의 기피 스타일은 탐색 후보에서 제외한다. |
| 입력 데이터 | 유저 선호 점수 테이블 레코드, 전처리된 날씨/기온 JSON, 유저 입력 TPO 텍스트, 옷장 보유 의류 배열, 자연어 수정 요청 텍스트(선택) |
| 출력 데이터 | 3\~4종의 독립 코디네이션 세트 배열(아이템 매핑 목록 + 자연어 추천 설명 포함) JSON |
| 입출력 유형 | 내부 데이터 연산 및 외부 API 통신 |
| 데이터 파일 유형 | 내부 파일(유저 취향 데이터베이스) |
| 중요도 | 상 |
| 위험도 | 상 |
| 품질속성 | 기능 적합성, 상호운용성, 사용성 |
| 평가 방법 | 1. Dynamic JSON 주입 시 메인 LLM이 지정된 기피 속성을 배제하는지 프롬프트 로그를 검증한다.<br>2. 최종 출력 스트림 데이터에 3\~4셋의 코디 구조와 자연어 사유(reason 필드)가 규격대로 파싱되는지 검사한다.<br>3. 각 코디의 추천 사유가 날씨·TPO·사용자 선호 근거를 구체적으로 포함하는지 출력 내용을 검토한다.<br>4. 자연어 수정 요청 입력 후 재생성된 코디가 해당 요청을 반영하는지 출력 결과를 확인한다. |

### 옷장 데이터 부족 시 에센셜 의류 보완 추천

| 항목 | 내용 |
| --- | --- |
| 요구사항 유형 | 기능 |
| 요구사항 개요 | 사용자의 옷장 데이터만으로 특정 상황(TPO)에 적합한 코디 구성이 불가능한 경우, 에센셜 의류 DB를 활용하여 보완 추천 아이템을 제안하고, 보유 의류와 추가 추천 의류를 구분 표시한다. |
| 요구사항 내용 | **1.1** 시스템은 등록 의류 총 개수가 5개 미만이거나, 필수 카테고리(상의/하의/신발) 중 어느 하나라도 미등록 상태에서 추천을 요청한 경우를 '의류 부족 상태'로 정의한다.<br>**1.2** 의류 부족 상태 판정 시, 에센셜 의류 DB(반팔 티, 블랙 슬랙스, 베이직 데님, 화이트 스니커즈, 블랙 로퍼, 베이지 가디건 등 범용 기본 아이템 포함)의 활용 비중을 80% 이상으로 상향하여 코디를 생성한다.<br>**1.3** 의류 부족 상태가 아니더라도 특정 TPO(결혼식 하객, 면접, 격식 있는 모임 등)에 필요한 아이템이 옷장에 없는 경우 에센셜 의류 DB를 보조 참고하여 추가 추천 의류를 제안한다.<br>**1.4** 추천 결과 화면에서 사용자 보유 의류([사용자 보유 의류])와 에센셜 DB 기반 추가 추천 의류([추가 추천 의류])를 시각적으로 구분하여 표시한다.<br>**1.5** 의류 부족 상태에서 추천 결과 출력 시 "옷장이 비어 있어 기본 에센셜 아이템 위주로 추천해 드렸습니다. 보유하신 옷을 더 등록하시면 맞춤 코디가 생성됩니다."라는 안내 문구를 노출하여 추가 업로드를 유도한다. |
| 입력 데이터 | 사용자 옷장 등록 의류 수 및 카테고리 현황, 에센셜 의류 DB 전체 레코드, TPO 컨텍스트 |
| 출력 데이터 | 보유의류 / 기본 옷 구분 라벨이 포함된 코디 세트 배열, 의류 부족 시 UI 안내 문구 |
| 입출력 유형 | 내부 데이터 연산(DB 조회 및 조합 로직) |
| 데이터 파일 유형 | 내부 파일(에센셜 의류 DB 테이블, 사용자 옷장 DB 테이블) |
| 중요도 | 중 |
| 위험도 | 하 |
| 품질속성 | 기능 적합성, 사용성 |
| 평가 방법 | 1. 의류 부족 상태에서 추천 요청 시 에센셜 DB 기반 코디가 정상 출력되는지 확인한다.<br>2. 추천 결과 화면에서 보유 의류와 추가 추천 의류가 각각 다른 레이블로 구분 표시되는지 UI를 검증한다.<br>3. 의류 부족 안내 문구가 정상 출력되고, 추가 업로드 페이지로 정상 연결되는지 확인한다. |

### 2단계 검증 구조 및 후처리 재생성 파이프라인

| 항목 | 내용 |
| --- | --- |
| 요구사항 유형 | 기능 |
| 요구사항 개요 | LLM이 생성한 코디 결과물에 대해 백엔드 Hard Rule 기반 1차 검증과 LLM Fashion Reviewer 기반 2차 검증의 2단계 후처리 파이프라인을 수행하며, 검증 실패 시 최대 2회 자동 재생성을 시도한다. |
| 요구사항 내용 | **1.1** [1차 검증 - Hard Rule] 백엔드 코드 레벨의 조건문(If-Else)으로 다음 항목을 검사한다: 상의 또는 아우터 누락 여부 / 신발 누락 여부 / 동일 카테고리 중복 착용 여부 / 기온 대비 계절 역행 아이템 포함 여부(예: 28도 이상에서 패딩 추천). 1차 검증 실패 시 즉시 LLM 재생성을 요청한다.<br>**1.2** [2차 검증 - Soft Rule] LLM Fashion Reviewer 역할: 1차 검증을 통과한 코디에 대해 별도의 프롬프트를 통해 TPO 적합성·스타일 일관성·색상 조화도·사용자 선호 반영 여부를 종합 평가하여 {"pass": true/false, "reason": "..."} 형태로 판정 결과를 리턴받는다.<br>**1.3** 2차 검증 실패 시 실패 사유(reason)를 다음 재생성 프롬프트에 피드백으로 포함하여 LLM이 오류 원인을 인지하고 수정된 코디를 생성하도록 유도한다.<br>**1.4** 재생성은 최대 2회를 초과하지 않으며, 2회 연속 실패 시 기본 무결성 코디 셋(에센셜 의류 기반)을 대체 출력하여 서비스가 중단되지 않도록 한다.<br>**1.5** 검증 및 재생성 전 과정에서 사용자 화면에는 로딩 인디케이터를 유지하여 UI 블로킹이 발생하지 않도록 한다. |
| 입력 데이터 | LLM 생성 코디 세트 배열(JSON), 전처리 환경 컨텍스트, 사용자 선호 정보 |
| 출력 데이터 | 2단계 검증을 통과한 최종 코디 세트 배열 또는 대체 기본 코디 세트 |
| 입출력 유형 | 내부 데이터 연산(규칙 검증) 및 외부 API 통신(LLM) |
| 데이터 파일 유형 | 내부 파일(Hard Rule 딕셔너리, Fashion Reviewer 프롬프트 템플릿) |
| 중요도 | 상 |
| 위험도 | 상 |
| 품질속성 | 기능 적합성, 신뢰성, 효율성 |
| 평가 방법 | 1. 신발 누락·상의 누락·패딩+고온 조합 등 오류 코디를 주입하여 1차 Hard Rule 검증이 정상 탈락 처리하는지 확인한다.<br>2. 결혼식 TPO에 크롭티, 미니멀 요청에 하와이안 셔츠 조합 등을 주입하여 LLM Fashion Reviewer가 pass: false로 판정하는지 검증한다.<br>3. 연속 2회 검증 실패 시 대체 기본 코디가 정상 출력되고, UI 블로킹 없이 로딩 인디케이터가 유지되는지 확인한다. |

### DAR-001 데이터 보존 및 가중치 백업

| 항목 | 내용 |

## 품질 요구사항

### 신뢰성 요구사항 (Reliability)

- **RER-001 (시스템 가용성 확보):** 시스템은 업데이트 및 정기 패치 시간을 제외하고 24시간 중단 없이 서비스를 제공해야 하며, 추천 코어 API 서버 장애 발생 시 사용자에게 시스템 에러를 노출하는 대신 '서버 일시 점검 안내 페이지'로 안전하게 리다이렉트하여 웹 서비스의 최소한의 신뢰성을 유지해야 한다.
- **RER-002 (2단계 예외 복구성):** 후처리 2단계 검증 파이프라인에서 연속 2회 탈락 실패가 발생하여 무한 루프 위험이 있을 경우, 즉시 에센셜 의류 기반 기본 무결성 코디 셋을 대체 출력하는 예외 안전장치를 적용하여 신뢰성을 확보한다.
- **RER-003 (날씨 API 장애 대응):** 외부 기상 Open API 응답 실패 시, 시스템은 마지막으로 성공적으로 수신된 날씨 데이터 또는 사전 정의된 기본값(현재 계절 평균 기온)으로 전처리를 수행하여 추천 파이프라인이 중단 없이 진행되도록 한다. 단, 날씨 데이터 부재 여부를 사용자에게 안내 문구로 표시하여야 한다.
- **RER-004 (추천 이력 데이터 무결성):** 코디 추천 완료 이벤트 발생 시 이력 저장 트랜잭션은 원자적(Atomic)으로 처리되어야 하며, 저장 실패 시 재시도 로직을 통해 이력 누락이 발생하지 않도록 한다.

### 사용성 요구사항 (Usability)

- **USR-001 (화면 반응성):** 사용자가 스마트폰, 태블릿, PC 등 다양한 디바이스 환경에서 시스템을 이용하더라도 화면 레이아웃 크기가 자동으로 조절되는 모바일 반응형 UI 뷰를 제공해야 한다.
- **USR-002 (사용자 에러 보완성):** AI 자동 태깅 시스템의 인식 정확도 한계를 보완하기 위해 사용자가 결과 카테고리 속성을 직접 클릭하여 손쉽게 정정할 수 있는 수동 드롭다운 콤보박스 UI를 필수 배치한다.
- **USR-003 (코디 추천 결과 직관성):** 추천된 3\~4개의 코디 세트는 각각 의류 아이템 이미지(또는 대표 아이콘), 스타일 태그, 자연어 추천 사유를 함께 표시하여 사용자가 한눈에 코디의 콘셉트와 선택 이유를 파악할 수 있도록 구성한다.
- **USR-004 (보유 의류와 에센셜 의류 구분 표시):** 추천 결과 화면에서 사용자가 실제 보유한 의류([사용자 보유 의류])와 에센셜 DB 기반 추가 추천 의류([추가 추천 의류])는 배경색 또는 레이블 뱃지 등의 시각적 요소로 명확히 구분하여 표시한다.
- **USR-005 (자연어 수정 요청 진입점 제공):** 추천 결과 화면에 텍스트 입력창을 항상 노출하여 사용자가 "좀 더 캐주얼하게", "검정색 비율 줄여줘" 등의 자연어 수정 요청을 언제든지 즉시 입력하고 새로운 코디 후보를 재생성 요청할 수 있도록 한다.

### 유지보수성 요구사항 (Maintainability)

- **MAR-001 (하이퍼파라미터 격리화):** 가중치 피드백 점수 연산 파이프라인의 기초 상수(선택 +3, 거절 -1, 수정 요청 ±2) 및 클리핑 임계치(최대 +30, 최소 -15), Time Decay 계수(0.8) 수치는 소스코드에 하드코딩하지 않고 외부 환경 구성 파일(.env)로 격리 분리하여 시스템 재배포 없이 동적 유지보수가 가능하도록 설계한다.
- **MAR-002 (룰 기반 필터링 딕셔너리 외부화):** 기온 조건별 의류 제거/우선 포함 규칙(Rule Dictionary)은 소스코드 내 하드코딩이 아닌 외부 설정 파일(JSON 또는 YAML) 형태로 관리하여, 계절 규칙 추가·수정 시 코드 변경 없이 설정 파일만 업데이트하면 즉시 반영되도록 한다.
- **MAR-003 (LLM 프롬프트 템플릿 버전 관리):** 메인 코디 생성 프롬프트 및 Fashion Reviewer 검증 프롬프트는 버전 관리 체계(v1.0, v1.1 등)를 적용하여, 프롬프트 개선 시 이전 버전과의 A/B 테스트 비교가 가능하도록 구조화한다.
